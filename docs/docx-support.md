# DOCX 支持范围

Research IDE 直接打开并保存项目中的 `.docx`。文件本身始终是文稿事实来源；编辑过程中使用的 HTML/ProseMirror 结构只是内存编辑模型，不要求用户另存为应用专用格式。

旧版二进制 `.doc` 通过受约束的 LibreOffice headless 往返转换提供同文件打开/保存体验，具体边界见 [旧版 DOC 支持](./legacy-doc-support.md)。

## 当前可编辑内容

| 类别 | 当前支持 |
| --- | --- |
| 文本 | 段落、标题 1–6、粗体、斜体、下划线、删除线、上下标、黄色高亮、行内代码、换行 |
| 科研排版 | 1/1.15/1.5/2 倍及 0.8–4 倍自定义行距、段前/段后间距、左/中/右/两端对齐、首行或悬挂缩进、左右缩进、常用字体、6–96 pt 字号、十六进制文字颜色 |
| 块结构 | 引用、代码块、水平分隔线 |
| 列表 | 有序列表、无序列表及受限的嵌套层级 |
| 表格 | 插入普通表格与表头，增删行列、合并/拆分单元格、删除表格；保存时重新生成表格样式 |
| 链接 | `http`、`https`、`mailto` 外部链接；拒绝含凭据或其他协议的链接 |
| 图片 | 内嵌 PNG、JPEG、GIF、BMP；单张不超过 10 MB、保存及重新打开时图片合计不超过 32 MB |

导入使用 Mammoth 从 OOXML 提取语义 HTML，HTML 经白名单清理后交给 Tiptap/ProseMirror。主进程另行读取受支持的直接段落属性，把行距、段前后、对齐和缩进恢复到受约束的编辑属性；Research IDE 生成的字体、字号和颜色使用可识别的 OOXML 字符样式，从而能在再次打开时恢复。所有数值、字体名和颜色都会经过范围或格式校验，不能借排版属性注入任意 CSS。第三方 DOCX 中通过主题、级联样式、条件表格样式或复杂语言字体设置继承的效果，Mammoth 不一定能与 Word 一样解析，保存时可能按编辑器中实际可见的受支持属性重建。

导入与保存采用相同的 32 MB 图片总量上限；单独保留 20 MB 正文文本预算，并为 base64 膨胀和结构开销设置 80 MB 的编辑树 JSON 与转换 HTML 信封上限。保存时也会执行完整的导入、限额及清理回读，因此应用允许保存的文件可以再次打开。超过限制时拒绝打开或保存，避免把巨量内容送入 renderer。外部图片关系会从仅用于转换的内存副本中剥离，不读取本地绝对路径或远程图片，普通网页链接仍会保留。保存使用 `docx` 从受验证的编辑树重新生成整个 OOXML 包。因此段落语义和受支持内容可以继续编辑，但原文件的样式表、精确分页、浮动定位、主题、节设置和其他不在表中的结构不保证原样保留。

## 兼容性处理

打开文件后，编辑器顶部的可折叠兼容性区域只显示与当前文件有关的信息：

- 普通 DOCX 也会说明保存将重新生成版式；首次执行有兼容性影响的保存需要确认；
- 修订、批注、脚注/尾注、页眉页脚、Word 公式、文本框/旧绘图、域（包括动态引用和交叉引用）、内容控件、SmartArt、外部关系及转换器报告的问题会产生需确认的警告；
- 包含宏、数字签名、嵌入 OLE/附件、`altChunk` 或 Word 编辑保护的文件强制只读，避免静默删除、绕过保护或使其失效；
- 加密、损坏、ZIP64、异常压缩包、超过 100 MB 或无法提取正文的文件会拒绝进入编辑器。

兼容性确认只表示用户接受所列重建影响，并不会解锁只读阻断项。遇到只读或需要保真处理的文件，应使用 Word 或 LibreOffice 编辑原件。

## 保存事务

每次保存按以下顺序执行：

1. 校验编辑树、链接、图片和大小限制；
2. 比较当前源文件与打开时记录的 SHA-256，若文件被其他程序修改则拒绝保存并要求重新加载；
3. 在内存中生成完整 DOCX，并再次检查 ZIP/OOXML 包且执行转换回读；
4. 在 `.research_ide/backups/<snapshot-id>/` 创建带校验和的“DOCX before save”快照；
5. 在源文件同目录独占创建临时文件，写入后 `fsync`，最后以 rename 替换同一个 `.docx`；
6. 删除遗留临时文件，并把新哈希记录为当前打开版本。

生成、校验、快照或临时文件提交任一步失败，保存都会报错。替换提交之前原 `.docx` 不会被改写；成功后可从项目快照恢复保存前版本。

## 尚未提供

- Word 像素级版式或完整 OOXML 往返保真；
- 修订的接受/拒绝、批注线程、脚注/尾注、引用域、目录域和 OMML 公式编辑；
- 宏、数字签名、OLE/附件、SmartArt、浮动文本框和复杂绘图编辑；
- 原样保留主题、样式表、节、页眉页脚、分页和打印布局；
- 完整复现第三方 Word 主题字体、样式继承、精确/最小值行距和复杂列表段落的所有级联排版；这些值会尽可能导入，并可能规范化为上述可编辑范围；
- 与 Microsoft Word 相同的协作、权限保护和插件生态。

这些限制意味着当前方案适合以正文、标题、列表、表格、链接和常见图片为主的科研文稿，不适合作为高保真排版或复杂 Word 自动化的最终处理工具。

## 开源组件

- Tiptap OSS / ProseMirror：MIT，用于编辑模型和界面；
- Mammoth：BSD-2-Clause，用于 DOCX 语义导入；
- `docx`：MIT，用于生成 OOXML；
- `sanitize-html`：MIT，用于导入 HTML 白名单清理；
- JSZip：可按 MIT 使用，用于检查和处理 OOXML ZIP 包；
- `@xmldom/xmldom`：MIT，用于按 XML 结构清理外部图片关系。

以上是主要 DOCX 组件的高层说明。实际发布仍需从锁定依赖树生成完整第三方许可证清单，并随构建产物保留各依赖要求的许可证文本和 notices。
